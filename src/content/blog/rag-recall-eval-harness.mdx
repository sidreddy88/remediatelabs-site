## The problem with vibes-driven RAG

After the fixes in Parts 1–3, my RAG pipeline felt better. The wrong incidents weren't showing up as context. The dedup gate was catching duplicates it used to miss. Things seemed to be working.

"Seemed" is the problem.

Every change I made was a guess about whether it helped. I'd tweak the indexed text, run a query manually, look at the scores, and decide it looked right. That's not a process — that's superstition. And it has a specific failure mode: a change that helps one query silently breaks three others I didn't think to check.

The fix is a repeatable eval: a fixed set of test cases that runs in seconds, passes or fails deterministically, and gives you a before/after number for every change you make to the pipeline.

---

## What recall@3 measures

The metric I care about is **recall@3**: for a given query, is the correct document in the top 3 results?

Not rank 1 — rank 1 is too strict for a retrieval system that's feeding an LLM. The LLM gets all three results as context. If the right incident is anywhere in the top 3, the LLM has the information it needs.

Recall@3 = (number of queries where correct doc is in top 3) / (total queries)

A score of 8/8 means every query in the test suite returns the correct incident somewhere in the top 3 results. A score of 6/8 means two queries fail — those are the cases to investigate.

This metric has two important properties:

1. **Binary per test case.** Either the right document is in the top 3 or it isn't. No partial credit, no threshold-tuning. This makes failures obvious.

2. **Aggregatable.** You can run the full suite in one call and get a single pass/fail number. Regression testing becomes trivial.

## Designing test cases at three angles

For each distinct bug, I write queries from three angles:

**Angle 1 — Exact wording**

Copy words directly from the error description. This is the easiest case: the query contains text that appears verbatim in the indexed document.

```python
("S3 NoSuchKey missing key copy delete", ["inc_001", "inc_002", "inc_003", "inc_004"])
```

**Angle 2 — Paraphrase**

Describe the same error in different words. This tests semantic generalization — does the embedding model know that "file not found when copying object" is the same concept as "NoSuchKey on missing key"?

```python
("file not found when copying object in S3", ["inc_001", "inc_002", "inc_003", "inc_004"])
```

**Angle 3 — Root cause**

Query for the underlying cause rather than the error message. The diagnosis agent sometimes comes in with a theory about what went wrong, not the raw error text.

```python
("S3 operation without existence check causes key not found", ["inc_001", "inc_002", "inc_003", "inc_004"])
```

Across all three bug types, this produces 9 recall tests. I also write precision and noise floor checks.

## Precision checks and noise floor checks

Recall tests ask: does the right document come back? Two other test types ask the complementary questions.

**Precision checks**: when I query for Bug A, does Bug B NOT appear in the top 3?

```python
# S3 query should not return the TYPEERROR incident
("S3 NoSuchKey missing key", ["inc_001"], ["inc_006"])
#                                           ^ must NOT appear
```

**Noise floor checks**: what happens when I query for something completely unrelated to anything in the corpus?

```python
("database connection pool exhausted", [])
# no incident should score above the noise floor
```

If an unrelated query returns a confident-looking result, the noise floor is too high and you have a false-positive problem.

## Pinning canonical IDs as named constants

Before writing the test suite, I added one piece of scaffolding: named constants for the canonical document IDs.

```python
# Canonical incident IDs — stable reference points for test assertions
S3_NO_SUCH_KEY_IDS = {"inc_001", "inc_002", "inc_003", "inc_004"}
SYNTAXERROR_ID     = "inc_005"
TYPEERROR_ID       = "inc_006"
S3_TEST_ID         = "inc_007"
```

Without this, tests look like:

```python
assert "inc_001" in result_ids  # what is inc_001?
```

With it:

```python
assert S3_NO_SUCH_KEY_IDS & result_ids  # is any S3 incident in results?
```

More readable, and decoupled from the specific ID values. If the corpus is rebuilt and IDs change, update the constants in one place.

## The full test suite

```python
import pytest
from app.services.rag import IncidentRAGService

# Canonical IDs
S3_NO_SUCH_KEY_IDS = {"inc_001", "inc_002", "inc_003", "inc_004"}
SYNTAXERROR_ID     = "inc_005"
TYPEERROR_ID       = "inc_006"
S3_TEST_ID         = "inc_007"


@pytest.fixture(scope="module")
def rag():
    svc = IncidentRAGService()
    yield svc


def top3_ids(results):
    return {r["incident_id"] for r in results[:3]}


# --- S3_NO_SUCH_KEY recall ---

def test_s3_exact_wording(rag):
    results = rag.search_incidents("S3 NoSuchKey missing key copy delete", n_results=3)
    assert S3_NO_SUCH_KEY_IDS & top3_ids(results), "S3 incident not in top 3"

def test_s3_paraphrase(rag):
    results = rag.search_incidents("file not found when copying object in S3", n_results=3)
    assert S3_NO_SUCH_KEY_IDS & top3_ids(results)

def test_s3_root_cause(rag):
    results = rag.search_incidents("S3 operation without existence check causes key not found", n_results=3)
    assert S3_NO_SUCH_KEY_IDS & top3_ids(results)


# --- SYNTAXERROR recall ---

def test_syntaxerror_exact(rag):
    results = rag.search_incidents("Failed to parse model JSON response", n_results=3)
    assert SYNTAXERROR_ID in top3_ids(results)

def test_syntaxerror_paraphrase(rag):
    results = rag.search_incidents("LLM returned malformed JSON", n_results=3)
    assert SYNTAXERROR_ID in top3_ids(results)

def test_syntaxerror_root_cause(rag):
    results = rag.search_incidents("JSON parse error from language model output", n_results=3)
    assert SYNTAXERROR_ID in top3_ids(results)


# --- TYPEERROR recall ---

def test_typeerror_exact(rag):
    results = rag.search_incidents("Cannot read properties of undefined reading publish_decision", n_results=3)
    assert TYPEERROR_ID in top3_ids(results)

def test_typeerror_paraphrase(rag):
    results = rag.search_incidents("undefined property access in publish_decision consumer", n_results=3)
    assert TYPEERROR_ID in top3_ids(results)

def test_typeerror_root_cause(rag):
    results = rag.search_incidents("classifyFields json parse error openai", n_results=3)
    assert TYPEERROR_ID in top3_ids(results)


# --- Precision: S3 query should not return unrelated errors ---

def test_s3_does_not_return_typeerror(rag):
    results = rag.search_incidents("S3 NoSuchKey missing key", n_results=3)
    assert TYPEERROR_ID not in top3_ids(results)

def test_typeerror_does_not_return_s3(rag):
    results = rag.search_incidents("Cannot read properties of undefined", n_results=3)
    assert not (S3_NO_SUCH_KEY_IDS & top3_ids(results))


# --- Noise floor ---

def test_unrelated_query_low_scores(rag):
    results = rag.search_incidents("database connection pool exhausted postgres", n_results=3, min_score=0.0)
    if results:
        assert results[0]["score"] < 0.60, f"Noise floor too high: {results[0]['score']}"
```

12/12. Every recall test passes, both precision checks pass, the noise floor check passes.

## Limitations worth naming

A few things this suite doesn't catch:

**Small corpus.** Seven incidents is enough to find the bugs we've already found. At 700 incidents, the precision checks become more important — more candidates means more opportunities for wrong documents to creep into the top 3.

**Query distribution.** These test cases were written by the same person who built the system. Real queries from a diagnosis agent in production will have phrasing I didn't anticipate. The suite is a regression guard, not a coverage guarantee.

**Score thresholds aren't tested.** The recall tests don't care whether the correct document scores 0.95 or 0.55 — only whether it's in the top 3. A retrieval system that returns everything in the top 3 would pass recall@3 with perfect scores. Score threshold tests need to be added separately.

**No end-to-end tests.** This suite tests retrieval in isolation. It doesn't verify that the retrieved context actually improves the LLM's diagnosis quality — that requires a separate eval on the full agent pipeline.

These limitations are fine for now. The suite catches the regressions I care most about: wrong documents appearing in context and right documents being silently excluded.
